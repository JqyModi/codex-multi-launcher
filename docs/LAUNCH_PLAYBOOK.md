# Codex 多开助手冷启动发布手册

本文档记录公网分发、账号协作、素材制作和反馈承接流程。目标是在不共享账号密码的前提下，把发布动作做成可复用、可确认、可追踪的小流程。

## 账号协作方式

首选方式是用户在本机浏览器或对应客户端中自行登录账号，Codex 只负责在已登录会话里辅助整理、填写和预览内容。

适用平台：

- Linux.do
- 微信公众号后台
- X
- Threads
- 小红书
- 知乎
- Reddit

协作边界：

- 用户不需要把账号密码、验证码、2FA 码发给 Codex。
- 遇到登录、验证码、扫码确认、风控验证、实名/安全确认时，由用户手动完成。
- Codex 可以在已登录页面中填写标题、正文、标签、链接，上传图片或视频，并检查预览。
- Codex 不会在未得到明确确认前点击最终发布、删除、付费推广、私信群发、评论群发等会造成公网影响的按钮。

## 账号授权与发布实施方案

Codex 不接收、保存或代管平台账号密码。公网发布采用“本机登录会话 + Codex 辅助填充 + 用户最终确认”的协作方式。

### 推荐授权流程

1. 用户在本机浏览器打开目标平台并完成登录。
2. 遇到扫码、短信验证码、2FA、安全验证、实名确认时，用户手动完成。
3. 用户确认当前页面已经进入可编辑状态，例如发帖页、文章编辑页或笔记发布页。
4. Codex 通过浏览器或桌面自动化辅助填写标题、正文、标签和链接。
5. Codex 上传本地素材，并检查预览、链接、图片和视频是否正常。
6. Codex 汇总即将发布的内容摘要，等待用户明确确认。
7. 用户确认后，Codex 才能点击最终发布按钮；如果平台风控再次要求确认，由用户手动完成。

### 可操作边界

Codex 可以做：

- 打开平台发布页。
- 粘贴标题、正文、话题、标签和链接。
- 上传本地图片、GIF、MP4。
- 调整排版顺序。
- 检查预览、下载链接、Issue 链接和素材是否加载。
- 保存草稿。

Codex 不主动做：

- 输入或保存账号密码。
- 处理验证码、扫码登录和 2FA。
- 未经确认点击“发布”“群发”“删除”“推广”“置顶”等公网影响按钮。
- 代替用户作身份背书式回复。
- 批量私信或批量评论。

### 平台执行方式

Linux.do：

- 用户先在浏览器登录 Linux.do。
- Codex 打开发帖入口，填写标题和 Markdown 正文。
- 优先上传 `docs/marketing-assets/hero-annotated.png` 和 `docs/marketing-assets/codex-workflow-demo.mp4`。
- 如果平台不接受 MP4，则改用 `docs/marketing-assets/codex-workflow-demo.gif`。
- 发布前检查：标题、落地页链接、GitHub 链接、Issue 链接、Release 链接、素材是否可见。

微信公众号：

- 用户登录微信公众平台后台。
- Codex 辅助创建图文草稿，填写标题、摘要、正文。
- 首图使用 `docs/marketing-assets/hero-annotated.png`。
- 正文中插入 `docs/marketing-assets/codex-workflow-demo.gif` 或上传视频素材。
- 发布前检查：封面、摘要、正文链接、GitHub Issues 反馈入口、公众号二维码是否合适。
- 公众号的群发/发布确认建议由用户手动点击，Codex 最多停在预览或草稿完成状态。

X / Threads：

- 用户在浏览器登录账号。
- Codex 填写短文案，附落地页链接。
- 素材优先使用 MP4；如果平台转码失败，改用主视觉图。
- 发布前检查：首句是否明确痛点，链接卡片是否正确，视频是否可播放。

小红书：

- 用户在浏览器或客户端登录账号。
- Codex 准备标题、正文、标签和图片/视频素材。
- 小红书更适合“痛点截图 + 操作视频 + 步骤说明”的形式，标题避免过度技术化。
- 若网页版上传能力受限，由用户手动上传素材，Codex 负责整理发布文案和检查最终预览。

知乎：

- 用户在浏览器登录账号。
- Codex 可辅助发布想法或文章。
- 长文优先复用公众号稿，短内容复用 Linux.do 的精简版本。
- 发布前检查：标题是否像问题解决方案，正文是否清楚说明边界，是否引导到 GitHub Issues。

Reddit：

- 用户在浏览器登录 Reddit。
- Codex 辅助选择合适 subreddit、填写标题、正文和链接。
- 优先发布为 text post，正文说明开源求测背景、边界和反馈入口；如果 subreddit 更适合 link post，则使用落地页链接。
- 素材优先使用 MP4 或主视觉图，视 subreddit 发帖规则而定。
- 发布前检查：subreddit 规则、是否允许 self-promotion、标题是否过度营销、链接是否可访问。
- Reddit 发布建议使用英文短版文案，并明确这是 early MVP / looking for feedback。

### 发布前确认模板

每次正式发布前，Codex 应给出如下确认信息：

```text
平台：
发布类型：帖子 / 文章 / 笔记 / 想法 / 推文
标题：
正文摘要：
附带素材：
包含链接：落地页 / GitHub / Issues / Release / 公众号
最终动作：发布 / 保存草稿 / 预览
```

只有用户明确回复“确认发布”或同等含义后，Codex 才能执行最终发布动作。

## 发布确认规则

以下动作需要用户明确确认后才能执行：

- 公开发布内容。
- 修改已发布内容。
- 删除内容或评论。
- 置顶、精选、加精、付费推广。
- 向用户、粉丝、群组或私信对象发送消息。
- 使用含账号身份背书的回复。

确认前，Codex 应提供：

- 平台名称。
- 即将发布的标题和正文摘要。
- 附带素材列表。
- 目标链接。
- 是否包含下载链接、Issue 链接、公众号二维码。

## 第一阶段目标

第一阶段不追求泛流量，重点验证真实需求和产品可用性。

建议目标：

- 10-30 次真实下载。
- 3-5 个有效 GitHub Issues。
- 2 个第三方 Provider 兼容反馈。
- 1-2 个完整用户流程反馈。
- 公众号获得第一批精准关注用户。

核心指标：

> 成功创建并启动 2 个以上独立 Codex Profile 的用户数。

## 首发顺序

1. Linux.do 首发。
   - 目标：获取高质量开发者反馈。
   - 语气：开源求测、真实痛点、明确边界。
2. 微信公众号跟进。
   - 目标：承接长期更新、教程和兼容记录。
   - 语气：正式发布说明 + 使用教程 + 反馈入口。
3. X / Threads / 小红书 / 知乎 / Reddit 分批发布。
   - 目标：扩展触达面，拿到不同平台反馈。

## 素材要求

首发至少准备两类素材：

- 主视觉截图：展示 Profile 列表、选中的 Profile、Provider、Base URL、Open 按钮，并用标注强调“独立 API 配置”。
- 20-40 秒流程视频或 GIF：创建 Profile、填写 Base URL/模型、生成启动器、打开 Codex。

素材制作原则：

- 不展示真实 API Key、个人路径、隐私项目名。
- 优先使用演示 Profile 和示例 Base URL。
- 视频节奏要快，聚焦核心路径，不展示无关系统窗口。
- 图片标注只解释关键价值，不堆功能说明。

## 反馈承接

优先引导用户提交 GitHub Issues，方便跟踪和复现。评论区和公众号留言可作为补充入口，但不应替代 Issues。

推荐反馈模板：

```text
- macOS 版本：
- Codex App 版本：
- 使用接口：OpenAI 官方 / 自建代理 / 第三方兼容接口
- Base URL 是否以 /v1 结尾：
- 是否成功创建 Profile：
- 是否成功打开独立 Codex：
- 遇到的问题：
- 希望增加的功能：
```

## 风险表达

避免：

- 破解多开
- 绕过限制
- 无限多开
- 兼容所有中转站
- 绝对安全

推荐：

- 多 Profile 隔离
- 多配置管理
- 独立 API Key / Base URL
- 第三方 Responses 兼容接口
- 本地保存，诊断不包含 API Key
