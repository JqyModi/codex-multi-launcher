# Configuration Schema

## 1. Files

Codex Profile Manager owns these app-level files:

```text
~/Library/Application Support/Codex Profile Manager/profiles.json
~/Library/Application Support/Codex Profile Manager/secrets.enc.json
~/Library/Application Support/Codex Profile Manager/master.key
~/Library/Application Support/Codex Profile Manager/logs/
```

Codex Profile Manager creates these profile-level files:

```text
~/.codex-profiles/<profile-id>/codex-home/config.toml
~/.codex-profiles/<profile-id>/codex-home/profile-manager-backups/
~/Library/Application Support/Codex Profiles/<profile-id>/user-data/
~/Applications/Codex Profiles/<Profile Name>.app
```

## 3. Profile Fields

### id

Stable slug generated by the app.

Rules:

- Lowercase letters, numbers, and hyphen.
- Unique within `profiles.json`.
- Never changes after creation.

### name

User-facing display name.

Rules:

- 1 to 40 visible characters.
- Can contain spaces.
- Used for launcher display name.

### status

Allowed values:

- `active`
- `disabled`
- `deleted`

MVP can hard delete profiles only after creating a backup. Soft delete is preferred.

### appearance.iconBackgroundColor

Per-profile identity color used by the generated launcher icon and the manager sidebar marker.

Rules:

- Stored as uppercase `#RRGGBB`.
- Defaults to `#34C759`.
- The running status badge remains state-colored; this field is only an identity color.

## 4. Provider Types

### official_openai

Used when the user wants a separate official OpenAI API key profile.

Expected generated Codex config:

```toml
model = "gpt-5.2"
model_reasoning_effort = "medium"
```

The API key is injected through `OPENAI_API_KEY` at launch time.

### third_party_responses

Used when the user configures a third-party OpenAI-compatible provider.

Expected generated Codex config:

```toml
model_provider = "proxy"
model = "gpt-5.2"
model_reasoning_effort = "medium"

[model_providers.proxy]
name = "My Proxy"
base_url = "https://proxy.example.com/v1"
env_key = "CODEX_PROFILE_DEV_PROXY_API_KEY"
wire_api = "responses"
```

## 5. secrets.enc.json

Plaintext shape before encryption:

```json
{
  "schemaVersion": 1,
  "secrets": [
    {
      "profileId": "dev-proxy",
      "providerId": "proxy",
      "envKeyName": "CODEX_PROFILE_DEV_PROXY_API_KEY",
      "secretType": "api_key",
      "value": "sk-..."
    }
  ]
}
```

Encrypted file shape:

```json
{
  "schemaVersion": 1,
  "algorithm": "aes-256-gcm",
  "createdAt": "2026-05-11T00:00:00.000Z",
  "payload": {
    "iv": "...",
    "authTag": "...",
    "ciphertext": "..."
  }
}
```

Security rules:

- `master.key` must use file mode `0600`.
- `secrets.enc.json` must use file mode `0600`.
- UI logs must show only redacted values such as `sk-...abcd`.
- Diagnostics must never export decrypted secrets.

## 6. Generated Launcher Metadata

Launcher `Info.plist` should use:

```text
CFBundleDisplayName = <Profile Name>
CFBundleName = <Profile Name>
CFBundleIdentifier = local.codexprofilemanager.<profile-id>
CFBundleExecutable = launcher
CFBundleIconFile = profile-icon
```

Launcher resources should include:

```text
Contents/Resources/profile-icon.icns
```

Generated launcher script must know:

- Profile ID.
- `CODEX_HOME`.
- `user-data-dir`.
- Provider env var name.
- Path to helper command or manager app for secret retrieval.

## 7. Generated Codex Config Rules

The app owns these keys in generated `config.toml`:

- `model_provider`
- `model`
- `model_reasoning_effort`
- `[model_providers.<provider-id>]`

The app may preserve these copied settings from an existing Codex config:

- `plugins`
- `marketplaces`
- `mcp_servers`
- `features`
- `projects`
- `notify`

Conflict handling:

- If preserving copied settings causes parse errors, regenerate a minimal config.
- Always backup before overwrite.
- Show a diff summary before destructive changes in post-MVP.
