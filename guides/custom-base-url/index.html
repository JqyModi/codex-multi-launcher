Codex Base URL · Responses API

为每个 Codex 窗口设置不同 API。

每个 Profile 可以使用独立 API Key、Base URL、模型和服务商名称，适合官方 OpenAI API、自建代理或真正兼容 Responses API 的第三方服务。

最重要的兼容要求

Codex 桌面版使用的能力不等同于普通聊天客户端。第三方服务商需要实际实现 OpenAI Responses API，仅声称“OpenAI 兼容”并不代表一定可用。

只支持 /chat/completions 的服务商可能无法工作。请向服务商确认是否支持 Responses API、流式响应和 Codex 所需的工具调用格式。

如何填写

  • 服务商名称：只用于本地识别，例如“公司代理”或“测试线路”。
  • Base URL：填写服务商给出的 OpenAI 兼容入口，常见形式以 /v1 结尾。
  • 模型：必须是接口实际接受的模型 ID，而不是网页展示名称。
  • API Key：填写服务商为该兼容接口签发的 Key。

API Key 保存到本地加密文件中，不写入启动脚本，也不会包含在“复制诊断”结果里。

获取模型与测试接口

“获取模型”会尝试读取服务商的 /models 返回结果。如果服务商没有提供该接口，仍可以手动填写模型 ID。

Provider 测试用于提前发现网络、鉴权和 Responses API 兼容问题。测试失败时允许继续生成，便于进一步排查，但不代表生成后一定能正常对话。

常见错误

401 / 403

通常表示 API Key 无效、权限不足、额度不可用，或服务商要求不同的鉴权格式。

404

优先检查 Base URL 是否多写或少写了 /v1，以及服务商是否真的提供 Responses 路由。

能获取模型但无法对话

/models 可用只代表能列出模型，不代表 Responses、工具调用和流式返回均兼容。

DeepSeek、Kimi、通义、智谱能否使用

取决于所选服务商是否提供 OpenAI Responses 兼容入口和正确模型 ID，不能只根据模型品牌判断。
